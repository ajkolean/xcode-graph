## Why Parity Testing?

### The Problem

When maintaining components in multiple frameworks, inconsistencies can easily creep in:
- Visual differences in styling or layout
- Behavioral differences in interactions
- API inconsistencies between implementations
- Diverging features over time

### Our Solution

**Parity testing** ensures that React and Lit implementations remain identical by:
- **Side-by-side visual comparison** in a single story
- **Shared test coverage** with identical interaction tests
- **Automated regression detection** via Chromatic visual testing
- **Consistent API surface** verified through TypeScript

### Benefits

✅ **Framework Flexibility** - Use React for React apps, Lit for framework-agnostic needs
✅ **Guaranteed Consistency** - Visual and functional parity verified automatically
✅ **Single Source of Truth** - Design system tokens shared across implementations
✅ **Comprehensive Testing** - Both implementations tested with identical scenarios
✅ **Future-Proof** - Easy to add new frameworks while maintaining parity

## How It Works

### The ParityComparison Component

Our parity tests use a custom `ParityComparison` component that displays both implementations side-by-side:

```tsx
<ParityComparison
  componentName="Button"
  reactComponent={
    <Button variant="default" size="default">
      Click me
    </Button>
  }
  litComponent={
    <LitButton variant="default" size="default">
      Click me
    </LitButton>
  }
/>
```

**Visual Layout**:
- **Left pane**: React implementation (gray border)
- **Right pane**: Lit implementation (purple border, highlighted)
- **Centered content**: Components displayed identically
- **Labels**: Clear identification of each implementation

**Styling**:
```
┌─────────────────┬─────────────────┐
│  React Button   │   Lit Button    │
│ (Original impl) │ (Web component) │
├─────────────────┼─────────────────┤
│                 │                 │
│   [Button UI]   │   [Button UI]   │
│                 │                 │
└─────────────────┴─────────────────┘
```

### Event Logging

Parity stories include event logging to verify behavioral consistency:

```tsx
const [eventLogger] = React.useState(() => createEventLogger());
const [logs, setLogs] = React.useState(eventLogger.logs);

const handleReactClick = () => {
  eventLogger.logReactEvent('click');
  setLogs([...eventLogger.logs]);
};

const handleLitClick = (e: CustomEvent) => {
  eventLogger.logLitEvent('button-click', e.detail);
  setLogs([...eventLogger.logs]);
};
```

**EventLogger Component** displays:
- Timestamp of each event
- Source (React or Lit)
- Event type and details
- Clear button to reset log

This helps verify:
- Both implementations fire events correctly
- Event payloads are consistent
- Timing and order match expectations

## Writing Parity Tests

### Basic Pattern

Every parity story follows this structure:

```tsx
import { Button } from '../components/ui/button';
import { LitButton } from '../components-lit/wrappers/Button';
import { ParityComparison } from './components/ParityComparison';

export const Default: Story = {
  args: {
    variant: 'default',
    size: 'default',
    children: 'Click me',
  },
  render: (args) => (
    <ParityComparison
      componentName="Button"
      reactComponent={
        <Button
          variant={args.variant}
          size={args.size}
        >
          {args.children}
        </Button>
      }
      litComponent={
        <LitButton
          variant={args.variant}
          size={args.size}
        >
          {args.children}
        </LitButton>
      }
    />
  ),
  play: async ({ canvas }) => {
    // Wait for Lit components to register
    await new Promise(resolve => setTimeout(resolve, 300));

    // Test both implementations identically
    const buttons = await canvas.findAllByShadowRole('button');
    expect(buttons).toHaveLength(2);

    // Verify both are accessible
    for (const button of buttons) {
      expect(button).toHaveAccessibleName('Click me');
    }
  },
};
```

### Key Elements

**1. Shared Args**
- Define args once, apply to both implementations
- Ensures identical props/attributes
- Changes update both sides simultaneously

**2. Shadow DOM Awareness**
- Use `findAllByShadowRole` instead of `findAllByRole`
- Queries penetrate Shadow DOM for Lit components
- See [Testing Guide](/story/introduction-testing-guide) for details

**3. Wait for Lit Registration**
```tsx
await new Promise(resolve => setTimeout(resolve, 300));
```
- Critical: Lit components register asynchronously
- Always wait before querying
- Prevents flaky tests and visual snapshots

**4. Test Both Implementations**
```tsx
const buttons = await canvas.findAllByShadowRole('button');
expect(buttons).toHaveLength(2); // One React, one Lit

for (const button of buttons) {
  // Test both identically
  expect(button).toHaveAccessibleName('Click me');
}
```

## Testing Strategy

### When to Use Parity Stories

**Use Parity Stories When**:
- ✅ Demonstrating visual consistency between frameworks
- ✅ Verifying API compatibility (props/attributes match)
- ✅ Testing event handling across implementations
- ✅ Showcasing the dual-implementation capability
- ✅ Creating baseline for visual regression testing

**Use Individual Component Stories When**:
- ✅ Documenting component-specific features
- ✅ Testing framework-specific behavior
- ✅ Showing advanced usage patterns
- ✅ Demonstrating composition with other components

## Shadow DOM Considerations

### Why Shadow DOM Matters

Lit components use Shadow DOM for:
- **Style encapsulation** - No CSS leakage
- **Composition** - Slot-based content projection
- **Standards compliance** - Built on web platform APIs

This requires special testing considerations.

### Testing Pattern

**Standard DOM Query** (React):
```tsx
// ❌ Won't work for Lit components
const button = canvas.getByRole('button');
```

**Shadow DOM Query** (Lit):
```tsx
// ✅ Works for both React and Lit
const button = canvas.findByShadowRole('button');
```

**Query Both Implementations**:
```tsx
// Find all buttons (React + Lit)
const buttons = await canvas.findAllByShadowRole('button');
expect(buttons).toHaveLength(2);

// Test each button
for (const button of buttons) {
  await userEvent.click(button);
  expect(button).toHaveAccessibleName();
}
```

### Common Patterns

**Querying Elements**:
```tsx
// Find all matching elements
const buttons = await canvas.findAllByShadowRole('button');
const inputs = await canvas.findAllByShadowRole('textbox');
const checkboxes = await canvas.findAllByShadowRole('checkbox');
```

**Interacting with Elements**:
```tsx
// Click both buttons
for (const button of buttons) {
  await userEvent.click(button);
}

// Type in both inputs
for (const input of inputs) {
  await userEvent.type(input, 'test value');
  expect(input).toHaveValue('test value');
}
```

**Accessibility Testing**:
```tsx
// Verify both implementations are accessible
for (const element of elements) {
  expect(element).toHaveAccessibleName();
  expect(element).toHaveAttribute('aria-label');
}
```

## Adding New Parity Stories

### Step 1: Create Component Implementations

Ensure you have both implementations:
- React: `src/components/ui/my-component.tsx`
- Lit: `src/components-lit/ui/my-component.ts`
- Wrapper: `src/components-lit/wrappers/MyComponent.tsx`

### Step 2: Create Parity Story File

Create `src/stories/MyComponent.stories.tsx`:

```tsx
import type { Meta, StoryObj } from 'storybook/internal/csf';
import { expect, userEvent } from 'storybook/test';
import { MyComponent } from '../components/ui/my-component';
import { LitMyComponent } from '../components-lit/wrappers/MyComponent';
import { ParityComparison } from './components/ParityComparison';

const meta = {
  title: 'Parity/My Component',
  component: MyComponent,
  parameters: { layout: 'padded' },
  tags: ['autodocs'],
} satisfies Meta<typeof MyComponent>;

export default meta;
type Story = StoryObj<typeof meta>;

export const Default: Story = {
  args: {
    // Component props
  },
  render: (args) => (
    <ParityComparison
      componentName="My Component"
      reactComponent={<MyComponent {...args} />}
      litComponent={<LitMyComponent {...args} />}
    />
  ),
  play: async ({ canvas }) => {
    await new Promise(resolve => setTimeout(resolve, 300));

    // Test both implementations
    const elements = await canvas.findAllByShadowRole('...');
    expect(elements).toHaveLength(2);
  },
};
```

### Step 3: Add Test Coverage

Include comprehensive tests:

```tsx
export const InteractiveTest: Story = {
  play: async ({ canvas, userEvent }) => {
    await new Promise(resolve => setTimeout(resolve, 300));

    const buttons = await canvas.findAllByShadowRole('button');

    // Test interaction on both
    for (const button of buttons) {
      await userEvent.click(button);
      expect(button).toHaveAttribute('aria-pressed', 'true');
    }

    // Test keyboard navigation
    await userEvent.tab();
    expect(buttons[0]).toHaveFocus();

    await userEvent.tab();
    expect(buttons[1]).toHaveFocus();
  },
};
```

### Step 4: Add Variant Stories

Showcase different variants:

```tsx
export const AllVariants: Story = {
  render: () => (
    <div style={{ display: 'flex', flexDirection: 'column', gap: '1rem' }}>
      {['default', 'primary', 'secondary'].map((variant) => (
        <ParityComparison
          key={variant}
          componentName={`${variant} Button`}
          reactComponent={<MyComponent variant={variant} />}
          litComponent={<LitMyComponent variant={variant} />}
        />
      ))}
    </div>
  ),
};
```

## Best Practices

### ✅ Do's

**Maintain API Consistency**
```tsx
// ✅ Good: Identical APIs
<Button variant="default" size="lg" disabled={false} />
<LitButton variant="default" size="lg" disabled={false} />
```

**Use Shared Design Tokens**
```tsx
// ✅ Both implementations use Panda CSS tokens
import { css } from '../styled-system/css';
```

**Test Both Implementations Identically**
```tsx
// ✅ Same test for both
for (const button of buttons) {
  await userEvent.click(button);
  expect(button).toHaveAccessibleName('Click me');
}
```

**Wait for Lit Components**
```tsx
// ✅ Always wait before querying
await new Promise(resolve => setTimeout(resolve, 300));
```

### ❌ Don'ts

**Don't Skip Shadow DOM Queries**
```tsx
// ❌ Won't find Lit components
const button = canvas.getByRole('button');

// ✅ Use shadow DOM queries
const button = canvas.findByShadowRole('button');
```

**Don't Test Implementations Differently**
```tsx
// ❌ Inconsistent testing
expect(reactButton).toHaveTextContent('Click');
expect(litButton).toHaveAttribute('label', 'Click');

// ✅ Test both the same way
for (const button of buttons) {
  expect(button).toHaveAccessibleName('Click');
}
```

**Don't Forget Accessibility**
```tsx
// ❌ Missing accessibility checks
const buttons = await canvas.findAllByShadowRole('button');

// ✅ Verify accessibility
for (const button of buttons) {
  expect(button).toHaveAccessibleName();
  expect(button).not.toHaveAttribute('aria-disabled', 'false');
}
```

## Troubleshooting

### Components Don't Match Visually

**Cause**: Different CSS or design tokens
**Solution**: Ensure both use Panda CSS tokens from `styled-system/`

### Lit Component Not Found

**Cause**: Component not registered or query timing issue
**Solution**:
1. Check Lit component is imported in `.storybook/preview.ts`
2. Increase wait time: `setTimeout(resolve, 500)`
3. Use `findByShadowRole` instead of `getByShadowRole`

### Events Not Firing

**Cause**: Event handler naming mismatch
**Solution**:
- React: `onClick`, `onChange`
- Lit: `onButtonClick`, `onInputChange` (custom events)
- Verify wrappers convert events correctly

### Tests Flaky in CI

**Cause**: Timing issues with Lit component registration
**Solution**:
1. Use `findAll` instead of `getAll` (waits for elements)
2. Increase Chromatic delay in `.storybook/preview.ts`
3. Add explicit waits in play functions
