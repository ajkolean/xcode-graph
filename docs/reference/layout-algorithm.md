## Phase 0: Current Layout Pipeline Map

**Input → Clustering:** When a new graph is loaded, the layout controller first groups nodes into **clusters**. Nodes with the same project (for local code) or same name (for external packages) are bundled into one cluster[\[1\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/cluster-grouping.ts#L16-L24). Each cluster is tagged with a type (Project for local code, Package for externals) and accumulates its member nodes[\[2\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/cluster-grouping.ts#L16-L25). After grouping, the code analyzes each cluster to assign internal metadata[\[3\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/cluster-grouping.ts#L34-L39). This analysis identifies **test nodes**, determines **anchor nodes** (entry points like apps/CLIs), distributes other nodes into **layers**, and assigns each node a **role**[\[4\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/cluster-analysis.ts#L34-L42)[\[5\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/cluster-analysis.ts#L62-L71). For example, in cluster analysis, apps are marked as anchors (role = Entry), frameworks/libraries with many dependents become InternalFramework or InternalLib, and tests get the Test role[\[5\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/cluster-analysis.ts#L62-L71). All these attributes (like isAnchor, layer, dependencyCount, etc.) are stored in cluster.metadata for each node[\[6\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/cluster-analysis.ts#L70-L79).

**Clustering → Layout:** The **LayoutController** then computes the positional layout. It starts by grouping and analyzing clusters (as above) and then computing a hierarchical layout using D3 force simulation[\[7\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/controllers/layout.controller.ts#L90-L98). The **graph-strata** module computes global dependency **strata**: each node gets a stratum (layer depth) via an SCC condensation DAG (so cycles collapse and dependents sit above dependencies)[\[8\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/graph-strata.ts#L4-L11)[\[9\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/graph-strata.ts#L226-L234). This yields a nodeStratum map (0 = top-level entry nodes, higher = deeper dependencies) and flags any **cycle nodes** (SCC size > 1)[\[10\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/graph-strata.ts#L13-L21). It also calculates each node's **fan-in** (how many others depend on it)[\[11\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/graph-strata.ts#L243-L252)[\[12\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/graph-strata.ts#L259-L267). Next, the layout builds a map of nodes to their cluster IDs[\[13\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L272-L280) and computes **cross-cluster edge weights** (how many edges connect each pair of clusters)[\[14\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L275-L282). Using the earlier cluster metadata, it also prepares a lookup of which nodes are anchors and how many internal deps they have[\[15\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L284-L292)[\[16\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L294-L300).

**Initial Positions:** All nodes are then converted into D3 simulation nodes (simNodes) with **deterministic initial coordinates**[\[17\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L313-L321)[\[18\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L319-L327). A hash of each node's ID (and cluster ID) seeds a pseudo-random spread, ensuring repeatable layout runs. Each node's starting position is a random offset within its cluster, plus a larger offset per cluster so clusters start spaced out. Meanwhile, each cluster gets an initial center position on a **spiral** around the origin[\[19\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L350-L359). (The current code spaces cluster centers using a golden-angle spiral with radius ~√i \* spacing, to avoid all clusters starting at one place[\[20\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L353-L361).)

**Forces & Physics:** The layout then defines a series of forces and runs a D3 force simulation to **relax** the positions:

- **Node-Node forces:** A **link force** connects directly related nodes with a target distance. Cross-cluster links are made longer (~3.5×) and much weaker (strength scaled 0.02×) to avoid collapsing clusters together[\[21\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L474-L483)[\[22\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L66-L74). A **charge force** (negative) repels all nodes, and a **collision force** prevents node overlap[\[23\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L470-L479)[\[24\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L481-L488).
- **Cluster forces:** Using _d3-force-clustering_, nodes sharing a clusterId experience a mild clustering force to pull them together (strength 0.25) with a minimum separation distance[\[25\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L70-L78). Two custom forces then adjust cluster positioning: a **cluster repulsion** force pushes cluster centers apart (strongly in X, weak in Y) to avoid overlap[\[26\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L394-L402)[\[25\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L70-L78), and a **cluster attraction** force gently pulls connected clusters together if they drift too far (activated beyond 300px separation)[\[27\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L403-L410)[\[28\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L73-L78). Together these aim to form "neighborhoods" of related clusters without merging them.
- **Strata (Layer) force:** Crucially, a vertical **stratum force** drives each node toward a Y position proportional to its dependency depth: forceY(y = stratum \* layerSpacing) with relatively high strength[\[29\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L494-L501). This enforces the **geological strata** metaphor: higher-level modules sit in upper layers, deeper dependencies in lower layers. A complementary **dependency hang** force ensures parent nodes stay above their dependencies by at least a gap (72px): for each edge A→B (A depends on B), it pushes B downward if it creeps above A[\[30\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L214-L223)[\[31\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L215-L223). These together uphold the rule "dependents above dependencies" in the layout.
- **Cluster interiors:** Inside each cluster, a custom **circular boundary** force keeps nodes within a fixed radius from the cluster center (preventing nodes from straying outside their cluster's bubble)[\[32\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/forces/cluster-boundary.ts#L19-L28). Additionally, a **radial distribution** force positions nodes by their importance: it computes a target radius for each node from the cluster center such that anchors and high-degree nodes stay near the center, and low-degree nodes drift toward the perimeter[\[33\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/forces/cluster-boundary.ts#L96-L105)[\[34\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/forces/cluster-boundary.ts#L109-L117). For example, anchors (entry points) get ~15% of cluster radius (near center)[\[34\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/forces/cluster-boundary.ts#L109-L117), whereas nodes with few connections might get ~85% (toward the edge). This "solar system" effect (central core vs. peripheral orbits) emerges during simulation as the radial force pulls each node inward or outward toward its target radius[\[35\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/forces/cluster-boundary.ts#L65-L73)[\[36\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/forces/cluster-boundary.ts#L86-L95). All these cluster-internal forces are initialized per cluster and applied at each tick of the simulation.
- **Global centering:** A weak global centering force keeps the overall graph near the canvas center[\[37\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L490-L497)[\[38\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L492-L501). In 3D mode, additional Z-axis forces bring semantic depth: nodes with higher _fan-in_ (many dependents = "core infrastructure") are pulled closer to the viewer, while higher _stratum_ (deeper dependency layers) push back into the distance[\[39\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L514-L522). A mild centering in Z prevents the whole graph from drifting infinitely along the depth axis[\[40\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L526-L534).

**Simulation & Convergence:** All forces are applied in a D3 simulation that runs for a fixed number of iterations (320 ticks)[\[41\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L90-L98). On each tick, cluster centers are recalculated as the mean position of their nodes (so clusters move as nodes move)[\[42\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L372-L381)[\[43\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L386-L394). After each tick's forces, a custom step recenters all nodes of a cluster to follow any shift in the cluster's center[\[44\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L552-L560)[\[45\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L560-L568) (this keeps clusters coherent if their center drifts). The cluster repulsion/attraction forces are also reapplied each tick (with decay as the simulation cools) to adjust cluster spacing dynamically[\[46\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L546-L554)[\[47\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L549-L557). By the end of 320 iterations, node and cluster positions have settled into a stable layout (no further animation is needed, since this controller runs the simulation to completion synchronously[\[48\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/controllers/graph-layout.controller.ts#L102-L111)[\[49\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/controllers/graph-layout.controller.ts#L114-L123)).

**Output → Rendering:** Finally, the layout returns a Map of **node positions** (relative to their cluster) and **cluster positions** (absolute in the world)[\[50\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L646-L655). Each cluster position includes the cluster's center (x,y) and its size (width/height diameter)[\[50\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L646-L655). Node positions are stored in cluster-local coordinates by subtracting the cluster center[\[51\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L660-L668). The rendering layer (GraphCanvas) composes these: for each node, it finds its cluster's world coordinates and adds the node's local offset to get the node's screen position[\[52\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/components/graph-canvas.ts#L669-L678)[\[53\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/components/graph-canvas.ts#L673-L681). Clusters are drawn as circles at their (x,y) with radius = width/2[\[54\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/components/graph-canvas.ts#L731-L739)[\[55\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/components/graph-canvas.ts#L734-L742), using a dashed border (different pattern for local vs external clusters)[\[56\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/components/graph-canvas.ts#L815-L823). Each cluster is labeled with its name and node count[\[57\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/components/graph-canvas.ts#L847-L856). Nodes are drawn as icons or dots at their composite position, typically color-coded by type. (Currently, cycle nodes are not given special treatment in rendering - they appear as normal nodes.)

**Dataflow Summary:** **Graph (nodes+edges)** → **Cluster grouping** (groupIntoClusters maps nodes to clusters)[\[1\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/cluster-grouping.ts#L16-L24) → **Cluster analysis** (analyzeCluster assigns roles, anchors, layers)[\[5\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/cluster-analysis.ts#L62-L71) → **Global strata** (computeNodeStrata gives each node a vertical layer)[\[9\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/graph-strata.ts#L226-L234) → **Force simulation** (computeHierarchicalLayout runs D3 forces with cluster neighborhood forces, radial distribution, etc.) → **Node & cluster positions** (node coords relative to cluster, cluster centers absolute) → **Canvas rendering** (draw clusters as "planetary systems" and nodes within them, aligned in horizontal strata bands).

## Phase 1: Diagnosing the "Clothesline" Layout Failure

In the current setup, large graphs (like Tuist's) sometimes degrade into a **"clothesline"**: clusters spread far apart along a horizontal line. This happens when forces that should separate clusters vertically aren't strong enough to overcome others, allowing clusters to drift sideways.

One key factor is the **anisotropic cluster repulsion.** In the config, cluster repulsion is deliberately scaled down 4× in the Y direction (clusterRepulsionYScale: 0.25)[\[28\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L73-L78). This means clusters strongly repel each other horizontally, but barely push each other vertically. The intent was to "let strata dominate Y"[\[58\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L75-L78) - i.e. vertical positioning should be governed by dependency layering, not random repulsion. However, in practice this can cause nearly all clusters to settle on the same stratum level, pushing outwards in X to avoid collisions (hence a long line of clusters at similar Y). Essentially, the repulsion force finds the path of least resistance by spreading clusters sideways, since upward/downward movement is weakly resisted by the scaled-down Y component.

Another contributor is **insufficient counter-forces to pull clusters back together.** Cross-cluster links were made intentionally weak: their force strength is only 2% of a normal link[\[22\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L66-L74). Thus, edges between clusters don't do much to keep related clusters adjacent. The cluster attraction force tries to form neighborhood groupings, but it only activates beyond a distance threshold of 300px[\[28\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L73-L78). If the clusters are already drifting gradually apart, they may remain just within that activation distance, never triggering a strong pull back together. In the "clothesline" scenario, many clusters had anchor nodes at stratum 0 (top layer), so they all hovered around the same Y coordinate due to the strata force. With vertical repulsion dampened, nothing spread them into multiple strata, and the weak cross-cluster links/attraction failed to re-cluster them horizontally. The initial spiral seeding also doesn't enforce any layering, so if the forces don't actively create vertical separation, clusters can end up roughly in a plane.

In summary, the **root cause** is the imbalance in forces: vertical stratification relies entirely on node-level forces (nodes aligning to layers) without a corresponding strong mechanism at the cluster level. Clusters can all crowd the top layer (especially if many have top-level anchors) and push away horizontally because: - The cluster repulsion force is **too low in vertical direction** to stagger them into multiple bands[\[58\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L75-L78). - The cluster attraction is **too weak or not engaged** to pull horizontally distant clusters back together (and its range cutoff can prevent it from acting in time)[\[28\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L73-L78). - Cross-cluster link forces are **very weak**[\[22\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L66-L74), so they don't enforce a tight coupling between dependent clusters. - No direct force anchors cluster positions in X, so with nothing to constrain horizontal drift, clusters keep sliding outward until repulsion balances out, often forming a long line.

This diagnosis suggests that simply tweaking constants might not fully solve it; a more structural approach is needed to organize clusters by layers (strata) and constrain their spread. We'll address this by introducing a cluster-level layout strategy and additional anchoring forces in subsequent phases.

## Phase 2: Architectural Layout with Dominant Strata (Cluster-Level Sugiyama)

To enforce the **"geological strata"** metaphor at the cluster level, we will introduce a deterministic layered layout for clusters _before_ the physics simulation fine-tunes positions. The idea is to treat each cluster as a node in a higher-level **cluster DAG**, arrange them in horizontal layers (strata) based on dependencies, then only allow minor adjustments from that structure.

**2.1 Build Cluster DAG:** First, derive a directed acyclic graph of clusters. We'll create a list of directed edges between clusters: if any node in Cluster A depends on a node in Cluster B, then add an edge A→B (meaning A requires B). This can be done by scanning all cross-cluster edges in the graph. Using this cluster DAG, perform a **layer assignment** similar to how computeNodeStrata does for nodes. Clusters with no other cluster depending on them (no incoming edges in the cluster DAG) will be top-level (layer 0, top strata band). If Cluster X depends (directly or transitively) on Cluster Y, then X's layer must be higher (numerically greater) than Y's. Essentially we compute longest-path lengths on the cluster DAG (treating cycles if any, though cluster cycles are rare - if present, they can be coalesced like SCCs). This gives each cluster a base **stratum level**.

**2.2 Initial Layered Positions:** Assign each cluster an initial coordinate based on its layer. For vertical placement, use a **fixed strata spacing** (e.g. 200px or a multiple of node layerSpacing to ensure distinct bands). For example, all clusters at layer 0 could start around y=0, layer 1 at y=+200, layer 2 at y=+400, etc. This deterministic layering ensures clusters are stacked in clear horizontal strata from the outset. Next, for horizontal placement within a layer, we will arrange clusters in a grid or ordered columns. We can use a **barycentric ordering** to reduce edge crossings: for each cluster, compute the average position of its neighbors from adjacent layers (e.g. the mean column index of its dependencies) and sort accordingly. This is akin to the Sugiyama algorithm's crossing minimization step. The result is that clusters in each layer will be ordered (left-to-right) in a way that aligns with how they connect to layers above or below, minimizing tangled inter-layer edges.

Using that order, distribute clusters evenly across X within that layer. For instance, if layer 2 has five clusters, we might position them at x = -400, -200, 0, 200, 400 (spaced by a constant or by cluster size). We will also respect the existing cluster size when spacing columns - larger clusters might need a bit more horizontal gap. This yields an initial **grid layout**: clusters neatly stacked in horizontal strata and roughly column-aligned where possible. (If a layer is too crowded, we may stagger rows slightly or increase spacing to avoid overlap.)

**2.3 Seeding and Soft Anchors:** We will inject these positions into the layout as the **initial cluster center coordinates**, replacing the current spiral seeding[\[19\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L350-L359). Each cluster's clusterCenterMap entry will start at (cx, cy) from our layered grid instead of the spiral. To preserve determinism, the ordering within each layer can be made deterministic (e.g., sort by cluster name or id when barycentric values tie, so it's stable). Now, to prevent the physics simulation from undoing this structured layout, we will add **soft anchor forces** on clusters. One approach is to create a custom force that nudges each cluster center toward its initial (layered) position. For example, we can apply a weak **forceX/forceY** on each node toward the cluster's target (cx, cy). Since we don't have cluster centers as independent simulation bodies (they are derived from node positions), the implementation can periodically adjust node velocities: e.g. at each tick, for each cluster, gently push all its nodes toward the cluster's target center. This effectively acts like a spring pulling the cluster back to its "grid" location if it strays.

Alternatively, we could treat one representative node per cluster (say the anchor node) as an attractor to the cluster's ideal position. But a simpler method is: after computing the initial layout, store each cluster's target (x0, y0). Then in the simulation tick loop, after updating cluster centers, compute a small delta for each cluster center relative to (x0, y0) and apply a portion of that delta negatively to the cluster's nodes (i.e. pull the cluster back toward the target). This keeps clusters loosely constrained around their designated layer-band location without making it rigid.

**2.4 Strengthen Vertical Lock-In:** Along with cluster-level anchoring, we will increase the strength of node-level vertical forces to truly lock layers in place. The current layerStrength (0.35) can be raised (e.g. ~0.5 or more) to make the forceY for strata more authoritative[\[59\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L79-L82). We can also introduce a slight **vertical clamping**: for example, limit how far a node can drift from its assigned stratum plane. This could be done by modifying the hang force or by an explicit clamp after layout - but a gentler method is to augment the hang or strata force to grow stronger with deviation. For instance, if a node is supposed to be at Y=240 (layer 2*120) but drifts ±60px, the force pulling it back could ramp up beyond that threshold. This ensures nodes (and thus cluster extents) don't blur across multiple strata too much. Each cluster might also compute its internal* _height_\* (difference between highest and lowest node). We could set a max height threshold per cluster layer (perhaps one strata step or so) and if a cluster's nodes try to exceed that, increase internal forces or apply a slight compression.

**2.5 Adaptive "Hang" for Key Edges:** To further enforce directional layering, we'll refine the **dependency hang force**[\[30\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L214-L223). Currently it nudges every dependent->dependency pair with equal strength. We can make this **adaptive**: for edges that cross between clusters or connect major components, apply a stronger hang (larger gap or force). For example, if an edge connects two clusters (i.e. a cross-cluster dependency), that relationship is critical for global architecture, so we want to clearly separate those clusters vertically. We can detect cross-cluster links (we already do for bundling); for each such link, perhaps multiply the hang gap by a factor (e.g. important cluster-to-cluster edges enforce a bigger vertical distance than internal ones). Also, if a cluster has many outgoing edges to dependencies below, we might give it a larger _effective_ layerSpacing. Concretely, if cluster A depends on cluster B and has, say, 10 edges into B (meaning heavy usage), we could treat the required vertical gap between A's layer and B's layer as larger. This could be done by dynamically increasing hangGap or applying multiple "push-down" iterations for those edges. The outcome: clusters that are tightly coupled via dependencies will still sit in adjacent strata, but with enough vertical separation that their linking edges slope visibly downward (no inverted or flat connections).

By implementing the above, **cluster strata will dominate the layout**: clusters start and remain in discrete horizontal bands. The physics simulation will then operate within these constraints - cluster repulsion can still adjust X spacing and slight Y tweaks, but the overall layered structure persists. Cross-cluster edges will now mostly connect only between adjacent strata (since the DAG layering ensures minimal jumps), and the risk of the "flat clothesline" (all clusters on one level) is eliminated by design.

## Phase 3: Cluster Interiors as "Solar Systems"

With clusters arranged in layers, we refocus on each cluster's **internal layout**, reinforcing the "solar system" metaphor. Currently, nodes within a cluster are arranged by a radial force towards a target radius computed from node degree (internal dependency counts)[\[33\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/forces/cluster-boundary.ts#L96-L105)[\[60\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/forces/cluster-boundary.ts#L114-L122). While this yields a basic center-periphery distribution, we will redesign it to account for the **semantic roles** of nodes (anchors, frameworks, tests, etc.), producing more meaningful "planetary" arrangements.

**3.1 Enhanced computeNodeTargetRadius:** We'll rewrite this function to incorporate node roles and cluster layer info, rather than just using raw degree. For example: - **Anchor nodes** (metadata.isAnchor == true, i.e. NodeRole.Entry) should sit almost at the cluster center (like the "sun"). In the new scheme, we can set anchors' target radius to a very small value (even 0 for a perfect center, though a small multiple of node size is safer to avoid overlap). We will preserve a minimum offset (similar to current 15% of radius) for anchors if multiple anchors exist. - **Internal Frameworks** (NodeRole.InternalFramework - typically core libraries or frameworks in the cluster) should also cluster near the center, just outside the anchors. These are the "big planets" that form the cluster's core. We might assign them a radius like ~25% of the cluster radius. In practice, we can map the cluster's internal layer (from cluster analysis) or role to a fraction of cluster radius. For instance, layer 0 (anchors) = 0%-15%, layer 1 (next important nodes, often frameworks) = ~30%, layer 2 (less connected libraries) = ~60%, and so on, up to outer ring for utilities. - **Internal Libraries (shared utilities)** - NodeRole.InternalLib or Utility - occupy intermediate orbits. If we have a cluster layer number from metadata (metadata.layer assigned in cluster analysis[\[61\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/cluster-analysis.ts#L58-L66)), we could directly use that. For example, a node with layer 2 might get radius ~60-70% of cluster radius, layer 3 nodes ~80-90%, etc. This aligns with cluster-analysis's intent: higher layer number meant more peripheral in the cluster's internal connectivity graph[\[62\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/cluster-analysis.ts#L182-L190)[\[63\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/cluster-analysis.ts#L194-L201). - **Test and Tool nodes** (NodeRole.Test and NodeRole.Tool) should be explicitly treated as **satellites on the periphery**. Often, test targets and CLI tools are not central to the cluster's functionality; they orbit around the main code. We will assign them a radius at or near the cluster's boundary. For instance, tests might get ~95% of cluster radius (almost touching the cluster edge). If needed, we can even let them exceed the normal radius slightly and rely on the boundary force to keep them just inside. This visually distinguishes tests/tools as separate "moons" orbiting the cluster. - Optionally, we can incorporate whether a test has a specific subject: e.g., if a test metadata.testSubjects exists linking it to a particular internal node[\[64\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/cluster-analysis.ts#L41-L49), we might try to place that test opposite or near that node on the circle. However, this level of detail might not be necessary in the general radial force - it could be handled by a post-processing adjustment.

In code, computeNodeTargetRadius(metadata, clusterRadius) will switch on metadata.role (or use booleans from metadata): - If isAnchor, return very small radius (e.g. 0.1 \* clusterRadius or a constant ~30px). - Else if role is InternalFramework, return something like 0.25 \* clusterRadius. - Else if role is InternalLib, maybe 0.4-0.5 \* clusterRadius. - Else if role is Utility, ~0.7 \* clusterRadius. - Else if role is Tool (CLI) or if metadata.hasExternalDependents is true but it's not anchor (meaning it's an external-facing utility), we might treat similar to InternalLib or Utility depending on its dependency count. - If role is Test, return 0.9 \* clusterRadius (very outer ring). - As a fallback (if metadata is missing or roles unrecognized), we can default to current behavior (degree-based radius ~50%).

This role-based approach ensures that even if a crucial framework has only moderate degree, it still stays near the center because we explicitly mark frameworks as important. Conversely, a test might have a higher degree (maybe it calls many functions) but will still be placed at the edge due to its role.

**3.2 Interior Layout Presets:** We recognize that clusters of different **types** might have different typical structures: - A **Project-type cluster** (local app module) often has an **app** anchor and possibly a couple of frameworks, many libraries/utilities, plus test targets. The solar system might look like a sun (app) in the center, big planets (frameworks) in inner orbits, numerous smaller modules further out, and test targets orbiting at the fringe. - A **Package-type cluster** (external dependency) might have no "app" anchor. Often it could consist of one or a few libraries or frameworks. In such cases, our identifyAnchors logic likely chose the most depended-upon library as an anchor anyway[\[65\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/cluster-analysis.ts#L100-L108)[\[66\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/cluster-analysis.ts#L115-L120). If so, the layout will treat that as center. If not (e.g. a package cluster with several equal libraries), we might allow multiple "anchor-like" centers or simply treat them all as inner layer nodes. For package clusters, an alternative preset could be a simple radial or even a linear layout inside, since these clusters might be smaller. - We can introduce special handling for clusters with multiple anchors (e.g., if a cluster had two apps). In that rare case, we might allow two central nodes (binary stars!), spacing them slightly apart at center by adjusting the collision radius. The radial force for anchors could push them apart a bit so they don't sit exactly overlapped at radius 0.

Implementing presets could mean providing alternate radius assignment logic based on cluster.type or cluster size. To keep it simple, we will primarily rely on role/layer as above, which inherently adapts to cluster content. But we will test on known scenarios (like Tuist graph) to fine-tune: e.g., if test nodes cluster too tightly at edges, we might spread them evenly around the circle via a small angular separation rule, or if anchors + frameworks all pile in center, we might enlarge anchor radius slightly when there are many inner nodes.

**3.3 Applying the Radial Force:** With the new target radius function, the **radial force** (createClusterRadialForce) will automatically pull nodes to those positions[\[35\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/forces/cluster-boundary.ts#L65-L73)[\[67\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/forces/cluster-boundary.ts#L68-L76). We might consider increasing the radialStrength a bit (currently 0.25) if needed, since we now trust our target placements more. However, too high strength can make the layout rigid; we likely keep it moderate and rely on collision/repulsion to finalize exact spots.

In addition, we should verify the **cluster boundary** force is compatible with our scheme. We might expand the maxRadius for a cluster if we expect tests at 0.95\*radius - currently maxRadius = clusterRadius - nodeRadius - 10[\[68\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L418-L425), which is fine. If tests or some nodes occasionally get pushed slightly beyond, the boundary force will nudge them back in, which is acceptable.

**3.4 Visual Outcome:** After these changes, each cluster drawn on the canvas will more clearly resemble a solar system: - **Anchor (entry)** at center with a bold icon (the sun). - **Core frameworks/libs** orbiting nearby (inner planets). - **Auxiliary utilities** further out (outer planets). - **Test and tool targets** on the outermost orbit (tiny moons). These might appear just inside the cluster's border or on it (with cluster border perhaps drawn dashed differently for projects vs packages as already done[\[56\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/components/graph-canvas.ts#L815-L823)). - Possibly multiple rings can be subtly visible (though we don't explicitly draw rings, the layering will be evident from node positions).

By prioritizing semantic roles in the physics, the layout communicates architecture: e.g., in a Tuist module graph, the central node in each cluster will be the app or primary module, surrounded closely by core internal frameworks, etc., giving an intuitive "center of gravity" for each cluster.

## Phase 4: Stabilizing Cluster Neighborhoods (Preventing Horizontal Drift)

To ensure the clusters remain well-arranged and **don't drift indefinitely in X** (or any direction), we need to introduce global constraints on cluster positions. The goal is to keep clusters relatively **clustered around a central vertical axis** (so the graph stays compact) while preserving the stratified Y layering.

**4.1 Vertical Band Anchoring:** One strategy is to create an implicit "column" down the middle of the graph and tether clusters to it. This doesn't mean forcing all clusters to x=0, but rather discouraging them from flying far off to the sides. We can implement a **weak centering force on cluster centers**. For example, define an ideal X-coordinate for each cluster (which could simply be 0 for all, or perhaps staggered columns if we want multiple columns). Given our Sugiyama layered layout, if some layers have many clusters, we might naturally have multiple columns of clusters; but we can still keep the _center of mass_ of each layer near x=0.

A practical approach: after we compute initial layered positions (Phase 2), we know the span of clusters in each layer. We can offset each layer so that its midpoint is at x=0 (centering that row). Then, during simulation, add a gentle **forceX** on each cluster's nodes toward x=0. This could be as simple as adding a d3.forceX(0) on the entire simulation with a very small strength for clusters (maybe scaled by cluster size). However, a plain forceX on all nodes might compress the whole graph horizontally, undoing some separation. So a refined method is: for each cluster center, measure its deviation from 0 and apply a corrective push. We could incorporate this into the cluster attraction force or as a separate pass: e.g., each tick, for each cluster center, if cx is say +800px, apply a small velocity to all nodes in that cluster toward negative X. Essentially a proportional controller pulling clusters back toward the y-axis.

Additionally, we ensure that **each strata band remains tight**: if one cluster in layer 3 drifts far left and another far right, the band is too spread. A possible solution is a _band-wise_ anchoring: compute the average X of all clusters in a given layer each tick, and nudge clusters toward that average. This would keep clusters of the same layer forming a more coherent row rather than isolated extremes.

**4.2 Expandable Bounding Radius:** Another way to limit drift is to imagine an invisible bounding circle that contains all clusters, and gradually increase its radius as the graph grows in size - but not let clusters escape beyond it. For instance, define R = k \* sqrt(N) where N is number of clusters or nodes, and k is some constant spacing factor. After the simulation, or periodically during it, if a cluster center's distance from origin > R, we clamp it or pull it inward. We can implement this as a **global boundary force** similar to cluster boundary but for the whole graph: if any cluster center is outside radius R, push it toward center.

We can initially choose k such that R is just large enough to fit the layered grid we computed. As a graph grows (N increases), R grows sublinearly (since layout area tends to grow roughly with sqrt(N) for dense packing). This prevents extreme outliers.

**4.3 Implementation in Code:** We will likely combine both methods subtly: - Add a new custom force in the simulation loop (Phase 5 of computeHierarchicalLayout). For each cluster center (cx, cy), do: - If |cx| is large, apply a small correcting acceleration ax = -ε \* cx on its nodes (ε a tiny constant). This is like a spring pulling cluster to vertical line. - If the radial distance sqrt(cx^2+cy^2) exceeds R (our threshold), apply a stronger inward pull on that cluster's nodes. - Another approach is to repurpose D3's forceCenter: currently they center nodes at (0,0) with strength 1.0[\[37\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L490-L497). We could reduce that strength for normal operation (so it doesn't interfere much) but rely on it for extreme cases. Alternatively, we set it to a moderate value (ensuring the graph's centroid stays near (0,0)). That alone might already mitigate drift, but careful: too high and it compresses the layout horizontally too much.

Given we observed "infinite X drift" specifically, we likely need an explicit check. We'll implement the **bounding circle** method as it directly caps the spread: - Calculate R after initial placement: e.g., if we have L layers and possibly M max clusters per layer, maybe set R = (M/2 \* clusterSpacing) + margin. Or simply find the furthest initial cluster's distance and use that as base R. - During simulation, for each cluster center beyond R, push it inward (similar to cluster boundary logic: compute the excess distance and subtract from velocity) - this is effectively the inverse of a repulsion. - As the simulation converges, ideally no cluster is beyond R. We can allow R to expand slightly if needed (maybe if collisions push clusters out, we give slack up to 10%). - Finally, after layout, we might have to re-center the entire set if they shifted (GraphCanvas fitToViewport already does a recentralization by looking at minX/maxX of cluster positions and centering in view[\[69\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/components/graph-canvas.ts#L32-L40)[\[70\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/components/graph-canvas.ts#L34-L42)).

By anchoring clusters in a vertical band and bounding overall radius, the **neighborhoods of clusters become stable**. We won't see one cluster flying off to far X-coordinates while others remain - they will tend to stay relatively aligned. This maintains a balanced, compact visualization (more like a **vertical stack** than a clothesline).

Notably, these constraints are gentle so that if the algorithm finds a genuinely better layout by offsetting some clusters (e.g., to avoid edge crossing), it can still do so within reason. We simply prevent unbounded drift.

## Phase 5: Semantic Depth (3D Z-axis Refinement)

In the 3D mode of the graph, we want the Z positioning of nodes (depth into/out of the screen) to correspond to **semantic layers** of the architecture. The current implementation already encodes some of this: nodes with higher _fan-in_ (meaning more components depend on them) are pulled closer to the viewer, and nodes in higher strata (deeper dependencies) are pushed further back[\[39\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L514-L522). We will refine this to better highlight Tuist's semantics: - **Core infrastructure forward:** Foundational libraries (which many modules use) should appear toward the front (positive Z), making them visually prominent. - **Feature modules mid-plane:** Ordinary feature modules or less critical components can sit around the middle. - **Apps and tests toward back:** Top-level apps (often at stratum 0) and test targets (which nothing else depends on) can be slightly recessed into the background, so the viewer looking "into" the graph sees the core first, with the outer shells (apps/tests) behind.

Currently, the formula is zTarget = log2(fanIn+1)\*80 - stratum\*25[\[71\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L518-L526), applied with strength 0.18[\[39\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L514-L522). For example, a core library (fanIn large, stratum maybe mid) could get a positive Z, and a high-stratum leaf (fanIn 0, stratum high) gets negative Z. But an app (fanIn 0, stratum 0) yields z≈0, neither forward nor back - yet we want apps to go _behind_ core libs. Tests similarly might have low stratum (possibly 0) and low fanIn, giving near 0 Z.

**5.1 Incorporate Roles into Z:** We will adjust the Z force to consider node roles or cluster layers: - If metadata.isAnchor (Entry, e.g., apps), we can treat it as if it had a **negative fan-in bias**. Perhaps subtract an extra term so that anchors get pushed back. For instance, we could add -200 to zTarget for anchors (making them appear behind by a fixed depth). Or multiply the stratum term for anchors (since many anchors are stratum 0 which normally gives no push, we want to artificially deepen them). - If role = Test or Tool, similarly push them back. Tests might even be given a special slight offset behind their subjects. But global effect: tests aren't central, so they can sit behind core logic. We could simply mark tests to have e.g. effectively stratum + 1 for Z calculation, or a small negative Z offset. - Conversely, if role = InternalFramework (core component), perhaps boost them forward. In many cases, these already have high fanIn, accomplishing this. But if an internal framework is only used by one cluster (so fanIn relative to cluster might be moderate), we might still want it forward because conceptually it's core. We could give all InternalFramework roles a +some Z bump. - InternalLib roles maybe small bump, Utility none or slight negative (since utility might not be heavily depended on).

Another angle: use the cluster's internal layer (0,1,2 from cluster analysis) as an indicator of core-ness: layer 0 (anchor) => back, layer 1 => forward (these are frameworks), layer 2 => mid, etc. If cluster metadata is accessible in layout (we have cluster.metadata per node), we can incorporate metadata.layer in the Z formula. For example: z += (someConstant) \* (idealLayerCenter - metadata.layer). If we consider anchor (layer 0) as idealLayerCenter maybe 2 (to back), framework (layer 1) as 0 (neutral or forward), util (layer 2+) as slight back again? We have to be careful; maybe simpler: subtract metadata.layer \* 20 from z (so higher layer number = further back). Anchors layer 0 subtract 0, tests have layer -1 perhaps but we treat tests separately. Actually tests in cluster metadata got layer -1[\[72\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/cluster-analysis.ts#L72-L79), we can map that to say layer 3 for Z push, or just detect role Test.

Probably easiest: implement explicit role-based Z adjustments: - Anchor/Test/Tool: add negative Z offset (e.g. -100 or -some factor \* clusterRadius to push them behind). - InternalFramework: add positive offset (+ something). - Others: no change (already handled by fanIn/stratum).

We will ensure these adjustments are small relative to the dynamic range to avoid extremes (since fanIn and stratum already produce a range).

**5.2 Clamp Z Spread:** The current scheme might produce Z values beyond a comfortable viewing range (depending on graph size). E.g., if a node had fanIn 50, log2(51) ~5.7, _80 ≈ 456, minus stratum maybe small, giving ~450 forward - possibly too far relative to the camera setup. Similarly a deep stratum could go - (e.g. stratum 5_ 25 =125 back). We should **clamp** the Z target to reasonable bounds. For instance, we might decide the front-most any node should get is +300 and back-most -300 (just an example - depends on camera). This can be done by min/max on the computed zTarget before applying the force. We also then adjust zCenterStrength and camera so that those extremes still appear in frame.

**5.3 Depth Force Implementation:** D3's forceZ is used to set a target z position per node[\[39\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L514-L522). We will modify the callback inside forceZ to our new function. Something like:

.force('meaningfulZ', d3.forceZ((d: any) => {
const meta = nodeMetadata.get(d.id);
let baseZ = Math.log2((fanIn.get(d.id) ?? 0) + 1) \* 80 - (nodeStratum.get(d.id) ?? 0) \* 25;
// Role-based tweaks
if (meta?.isAnchor || meta?.role === 'entry' || meta?.role === 'tool' || meta?.role === 'test') {
baseZ -= 100; // push anchors/tests/tools back
}
if (meta?.role === 'internal-framework') {
baseZ += 50; // pull core frameworks forward
} else if (meta?.role === 'internal-lib') {
baseZ += 20;
}
// Clamp
baseZ = Math.max(-300, Math.min(300, baseZ));
return baseZ;
}).strength(CONFIG.zStratumStrength));

(We will replace 'entry'/'internal-framework' etc with actual enum or constant values from NodeRole for a robust check.)

We also keep the existing weak centering .forceZ(0) for drift correction[\[40\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L526-L534). Possibly, we might strengthen zCenterStrength a tad if we find clusters drifting too far in depth (currently 0.02, maybe okay).

**5.4 Visual Effect:** After this change, in a 3D view of the graph, the viewer will notice that **core modules pop out** closer to the screen, while the outer applications and tests recede. If you rotate the graph, you'd see the strata now also aligned roughly in depth: entry points tend to lie on a "far" plane, foundational libraries on a "near" plane. This adds another visual cue to importance. We will fine-tune the constants with the Tuist data: for example, if apps are not clearly behind frameworks, we'll increase that anchor offset. If everything clusters too tightly in Z, we might widen the difference (increase fanIn multiplier or anchor offset). The clamp prevents any single outlier from flying off the depth range.

In summary, Phase 5 ensures the third dimension is used to reflect **architectural gravity**: high-gravity nodes (lots of dependents) sink toward the viewer (front), while low-gravity ones float toward the back.

## Phase 6: Highlighting Cycles in the Graph

Cycles (circular dependencies) are critical to visualize prominently ("cycles scream"). The engine already identifies all nodes that are part of a cycle (cycleNodes set in computeNodeStrata result)[\[10\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/graph-strata.ts#L13-L21). We will leverage that to adjust rendering and possibly layout: - **Visual Emphasis:** We will draw cycle-involved nodes with a distinct style. For example, a **glow or pulse effect**: We can add a neon-like outline or a continuous slow pulse (maybe via periodically changing opacity or size). Since the GraphCanvas currently uses canvas drawing, we can implement a pulse by altering the alpha or radius in each frame for those nodes. Another approach is to draw a **halo** circle around cycle nodes. A practical plan: in renderNodes, after drawing nodes normally, iterate through any node in cycleNodes. Draw an additional circle around it (maybe 1.5× the normal node radius) with a bright color (e.g., orange or red) and slight opacity, possibly using ctx.setLineDash(\[2,2\]) to make it flicker if rotating.

If using a timed pulse, we could utilize this.time (which we have in the render loop[\[73\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/components/graph-canvas.ts#L615-L622)). For instance, vary the glow radius or opacity with Math.sin(time/500) for a subtle pulse. We must ensure this doesn't clutter performance with too many strokes - but usually the number of nodes in cycles is limited (if cycles are few).

Alternatively or in addition, we could change the node shape or icon for cycle nodes (like add a warning icon or a knot symbol), but a glow/pulse is likely more universal.

- **Cycle Edges:** We should also highlight edges that are part of cycles. An edge is part of a cycle if both its source and target belong to the same strongly connected component of size > 1. We can identify this by comparing nodeSccId for the source and target (we have nodeSccId from strata analysis[\[74\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/graph-strata.ts#L14-L21)). If they match and that SCC size > 1, then this edge is in a cycle. In rendering, such edges could be drawn with a thicker stroke and/or a different color. For example, draw cycle edges in a bold dashed red line on top of the normal edge line.

Implementation-wise, in renderEdges, when iterating edges, check if (cycleNodes.has(edge.source) && cycleNodes.has(edge.target) && nodeSccId.get(source) === nodeSccId.get(target)). For those edges, set ctx.strokeStyle = 'red' (or a highlight color) and maybe ctx.lineWidth \*= 1.5 or ctx.setLineDash(\[5,3\]) to draw attention. We might draw them last so they overlay other edges.

- **Interactive Indication:** If the UI has a legend or on hover, we might want to indicate cycles. Possibly when a cycle node is hovered, we could highlight all nodes in that cycle. But for now, the above visual cues should suffice. We will document this so users know glow = cycle.

We do not plan on altering the **layout** algorithm for cycles beyond what's done (they already ensure cycle nodes share the same stratum to avoid paradox in layering[\[75\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/graph-strata.ts#L6-L14)). However, one tweak: cycles often cause the strata assignment to place all cycle nodes at the same depth. If a cycle is large, internally we might still want to spread those nodes a bit to avoid a tight knot. Currently, nothing explicitly clusters cycle nodes together aside from general forces. We can consider adding a slight **cycle repulsion** within an SCC to prevent them overlapping. But since they likely share cluster(s) and our normal collision covers it, this might be unnecessary.

So, deliverable for Phase 6 is primarily **rendering changes**: - Use the existing cycleNodes set (which we should now propagate from the layout result). (We might modify our LayoutController to include cycleNodes and nodeSccId in its return, or keep it globally accessible.) - Draw cycle nodes distinctly (glow/pulse). - Draw cycle edges distinctly (thicker/different stroke).

Once implemented, **cycles will visually "scream":** a developer glancing at the graph will immediately spot them via the special highlighting, prompting further inspection.

## Phase 7: Scalable Edge Bundling

The edge bundling feature groups cross-cluster edges into spline bundles for clarity, but it's currently disabled for large numbers of edges (if > 500 cross-cluster edges, bundling is skipped)[\[76\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L602-L609). For a graph as large as Tuist's, we want bundling to still assist readability without blowing up computation time. We'll improve bundling in two ways:

**7.1 Dynamic Budgets:** Instead of a hard cutoff at 500 edges[\[76\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L602-L609), we'll allow bundling for larger graphs by scaling down the work done. The bundling algorithm (likely using **force-directed edge bundling**) has parameters: number of cycles (iterations of subdivision) and iterations per cycle. Currently these are fixed (bundlingCycles: 5, bundlingIterations: 80)[\[77\]](https://github.com/ajkolean/xcode-graph/blob/b01ec8052a416ee6ee0c4eae4694b35b0f3cd870/src/graph/layout/d3-layout.ts#L102-L107). For 1000 edges, that full budget might be too slow. We can define a heuristic: e.g., target a constant budget of "operations" proportional to 500_80 = 40k edge-iterations (since 500 edges \* 80 iterations was deemed ok). So if we have E edges: - If E <= 500, do full 5 cycles \* 80 iterations. - If E is larger, we can reduce cycles and iterations such that E \* iterations stays around 40k-50k. For instance, if E = 1000, we could halve iterations (maybe 5 cycles \* 40 iterations = 200 iterations total, 1000_40 = 40k). If E = 2000, perhaps 3 cycles \* 30 iterations (~90 iterations total, 2000\*30 = 60k, a bit more but still manageable). Essentially, as E grows, we cut down iterations more aggressively, possibly also reduce cycles.

We'll implement this by computing factors: For E > 500, let ratio = E/500. Then set I = bundlingIterations / ratio (rounded down) and maybe C = bundlingCycles (keep cycles same unless extremely large E where we drop a cycle or two). Ensure minimum 1 cycle. For example: 1000 edges => ratio=2, I=80/2=40; 2000 edges => ratio=4, I=80/4=20; 5000 edges => ratio=10, I=8 (very low, maybe we also drop to 3 cycles to avoid diminishing returns).

This way, bundling still runs but with fewer iterations on large sets.

**7.2 "Lite" Bundling Option:** We will also expose a configuration option or automatically enter a **lightweight bundling mode** beyond a threshold. In lite mode, we might: - Use a higher compatibilityThreshold (meaning fewer edges are considered similar enough to bundle, which reduces computations). - Or use a simplified bundling: e.g., only bundle edges at a coarse level (like group edges by cluster pair and draw a single spline for each cluster pair connection).

A quick implementable idea: If E is huge (say > 2000), skip the iterative algorithm and do a **single-segment bundling**: for each cluster A→B combination, draw one curved link from A's center to B's center (perhaps as a quadratic Bezier or polyline with one control near midpoint). This won't reflect individual edges but gives an overview of cross-cluster connections. This could be a fallback if performance is critical.

More moderately, we can reduce bundling detail: reduce bundlingCycles from 5 to 3 for big graphs, so the curves are less smooth but still bundled.

We'll incorporate these adjustments into the bundling call. The pseudocode:

let maxEdgesForFull = 500;
if (crossClusterEdges.length > 0) {
let cycles = CONFIG.bundlingCycles;
let iterations = CONFIG.bundlingIterations;
if (crossClusterEdges.length > maxEdgesForFull) {
const ratio = crossClusterEdges.length / maxEdgesForFull;
cycles = Math.max(3, Math.floor(CONFIG.bundlingCycles)); // maybe keep cycles or reduce by ratio?
iterations = Math.max(10, Math.floor(CONFIG.bundlingIterations / ratio));
// Possibly adjust compatibility threshold or K as well if needed
}
bundledEdges = forceEdgeBundling(nodeMap, crossClusterEdges, {
K: 0.1,
C: cycles,
I_initial: iterations,
compatibility_threshold: CONFIG.compatibilityThreshold, // maybe raise threshold for large E
});
}

We remove the outright length < 500 check, so bundling always attempts at least minimal bundling if any cross-cluster edges exist.

**7.3 Performance Considerations:** We will benchmark on the Tuist graph: measure how long bundling takes after layout. The dynamic scaling should keep it manageable (on ~1000 edges, bundling with 40 iterations is likely okay). If needed, we'll adjust parameters. We'll document a configuration toggle to disable bundling entirely if users prefer speed.

The **visual effect** remains: even in larger graphs, cross-cluster edges will not appear as an overwhelming hairball. Instead, they'll bundle into cleaner conduits. With fewer iterations, the bundles might be less tight or smooth, but still far better than raw straight lines. We maintain "cross-cluster-only bundling" as per original (we do not bundle within a cluster, since those are usually fewer and clearer).

We will also be careful that bundling doesn't break determinism - the bundling algorithm should be deterministic given the same input (assuming no random initialization in it). If it does involve randomness, we'll see if we can seed it or otherwise ensure stable output (for the deterministic test in Phase 8).
